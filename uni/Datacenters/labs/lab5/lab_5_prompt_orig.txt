Create an interactive data center lab for students learning spine-leaf network topology. The lab must include two deliverables in one response: a static topology diagram and a live traffic simulator. Follow these exact specifications.— DIAGRAM —Draw an SVG topology diagram with:  • 2 spine switches (label: Spine 1, Spine 2 — role: L3 / ECMP)  • 4 leaf switches (label: Leaf 1–4 — role: ToR switch)  • 8 servers, 2 per leaf (label: Srv 1–8)  • Every leaf connected to every spine (full mesh uplinks)  • Servers connected only to their local leaf (downlinks)  • Three clearly labelled dashed-border zones: "Spine layer", "Leaf layer", "Servers / endpoints"  • A note that reads: "Always exactly 2 hops between any two servers"  • A legend identifying spine, leaf, and server node types by color  • All nodes clickable — clicking any node calls sendPrompt() asking Claude to explain that device's roleUse color coding: spines in purple (c-purple), leaves in teal (c-teal), servers in gray (c-gray).— SIMULATOR —Build an interactive HTML widget (not a static SVG) immediately below the diagram with:Controls:  • A "Source" dropdown listing all 8 servers with their leaf in parentheses  • A "Destination" dropdown listing all 8 servers  • A "Send traffic" button  • A "Clear" buttonSVG canvas inside the widget showing the same 2-spine / 4-leaf / 8-server topology. Each link is a line element with a distinct id so it can be highlighted independently.Simulation logic (JavaScript):  • Servers on the same leaf → highlight only the two server downlinks and the shared leaf; log "local switching, no spine needed"; show hop count as "0 (local)"  • Servers on different leaves → pick a spine using round-robin alternation (Spine 1 first, then Spine 2, alternating each click) to simulate ECMP load balancing; highlight source downlink, source-leaf-to-chosen-spine uplink, dst-leaf-to-chosen-spine uplink, dst downlink; activate all traversed nodes  • Highlight color for active path: teal (#1D9E75), stroke-width 3; idle links: tertiary color, 0.5px, opacity 0.35  • Clicking "Clear" resets all highlights and the logStats row (3 metric cards below the SVG):  • "Hop count" — shows 0 (local) or 2  • "Spine chosen" — shows "None (local)" or "Spine 1" / "Spine 2"  • "Traffic flows sent" — increments with each clickLog panel (monospace, muted background):  • Shows the full path as: Srv X → Leaf A → Spine N → Leaf B → Srv Y  • Explains ECMP selection on cross-leaf flows  • Notes when same-rack traffic bypasses the spineMake simulator downloadable as html file— LAB GUIDE (in response text, not inside the widgets) —Write a student lab guide with three parts:Part 1 — Draw the topology (15 min)  Students draw a fabric with 2 spines, 5 leaves, 2 servers per leaf on paper. Include a rule checklist and one discussion question about what happens to uplink count when a third spine is added.Part 2 — Trace traffic paths (10 min)  Provide a table with 4 pre-defined server pairs. Students use the simulator to fill in "Spine used" and "Path" columns. Include two discussion questions: one about local (same-rack) traffic and one about ECMP alternation across multiple flows.Part 3 — Failure scenario (5 min)  Ask students to cover Spine 1 on their drawn diagram and answer 3 written questions about connectivity, capacity loss, and whether full any-to-any reachability is preserved.End the lab guide with a "Key concepts" section defining: ECMP, ToR (Top of Rack), oversubscription ratio, and why spine-to-spine links are forbidden in this topology.
